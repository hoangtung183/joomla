Joomla: Adding a Filter to an Admin Table

What’s Being Done…

The objective is to add a simple filter to a Joomla 1.6 installation.

The use case described here is a Admin Table of “road races” containing descriptions of the various road race details. A filter is being added to distinguish the road races that have been publicly published to the site and which have not yet been published.

Credit: this blog post derives from the documentation provided at the Joomla docs wiki: How to add custom filters to component admin.

Note: the description here derives from real work on a inherited, legacy codebase. This is mentioned as it may account for discrepanies between how an ideal Joomla setup would work and what is described below.

Synopsis

  • The files of interest are in administrator/components/com_roadraces
  • Modify default.php to include the filter control
  • Add a populateState() method to models/roadraces.php to set the form value into the Joomla state
  • Modify the getListQuery() method to that Joomla state and modify the SQL query appropriately
  • Add the necessary Javascript to ensure the filter control correctly displays and submits the filter value

What is the Joomla component of interest?

For all the text below, we’ll assume our Joomla component name for which the Admin table is built is called “RoadRaces”.

Where are the files of interest?

The directory administrator/components/com_roadraces within the installation should house all the files of interest.

Where is the component data coming from?

models/roadraces.php contains the JModelList derived class which has the getListQuery() method which returns the data to the admin table.

This is what we’ll want to apply the filter to, likely using a SQL WHERE clause on the data query.

Where do the results of that query get mapped into the user-visible HTML table column names?

The file views/roadraces/tmpl/default_head.php outputs the column names for the header of the HTML table. The sibling file default_body.php outputs the rows of the table. These two files are invoked via the default.php file in that same directory.

(Note that these files may pipe through the JText::_() method for localization. The localization strings are located in administrator/language/. For simplification of this blog post, we do not localize the strings.)

Where should the filter control be added?

The views/roadraces/tmpl/default.php contains the host PHP file which builds the table. Assuming you want to insert the HTML control above the table, this would be a good place to insert the HTML for the control. If you want the filters to be part of the column headers themselves, then default_head.php would be the place to look.

Below is the base HTML for the drop-down filter. This is purely static control that is not yet functional.

        <!-- Published Filter -->
        <div>
                <h3>Filters</h3>
                <span>
                        <label>Published</label>
                        <select>
                                <option>Any</option>
                                <option>No</option>
                                <option>Yes</option>
                        </select>
                </span>
        </div>

How does this HTML filter communicate with getListQuery()?

The JModelList derived class in com_roadraces/models/roadraces.php has inherits a method called populateState from JModelList. This function has the opportunity to read any form-submitted data from the form created in the view’s default.php and put it into the Joomla state object.

Once the filter data is put into the state by populateState, this filter data can be used by getListQuery() to filter the search.

The following is added to the JModel derived class:

protected function populateState($ordering = null, $direction = null)
{
    // Initialise variables.
    $app = JFactory::getApplication('administrator');

    // Load the filter state.
    $published = $this->getUserStateFromRequest($this->context.'.filter.published', 'filter_published');
    $this->setState('filter.published', $published);

    // Be sure to call the default
    parent::populateState();
}

Note that the populateState implementation needs to call the inherited parent method as well.

How does the state filter the getListQuery() SQL query?

We assume the query is already constructed using a Joomla DBO query, a.l.a.:

    // Create a new query object.
    $db = JFactory::getDBO();
    $query = $db->getQuery(true);

    ...

So we then add in a filter if one has been set in the state by the populate method:

    // Account for filters
    $filterPublished = $this->getState('filter.published');
    if (!empty($filterPublished))
    {
        $query->where("race.published = $filterPublished");
    }

Enabling the HTML Filter to pass the filter info to the server

We’ll assume jQuery is available on the website:

        <!-- Published Filter -->

        <div>
                <h3>Filters</h3>
                <span>
                        <label>Published</label>
                        <select id='filter-published' name='filter_published'>
                                <option value=''>Any</option>
                                <option value='0'>No</option>
                                <option value='1'>Yes</option>
                        </select>
                </span>
        </div>
        <script type='text/javascript'>
        <!--
                $(function() {
                        //
                        // Set up the "Published" filter's current value and auto-submit on change
                        //
                        var filterPublished = "<?php echo $this->get('State')->get('filter.published') ?>";
                        $("#filter-published").change(function() {
                                this.form.submit();
                        }).val(filterPublished);
                });
        -->
        </script>

The additions are:

  • Adding a name attribute to the form element so that it will be accessible to the populateState method
  • Adding value attributes to the option elements corresponding the filter to use internally (rather than the plain English text)
  • Add some simple jQuery to set the form to the current value and auto submit on control changes
